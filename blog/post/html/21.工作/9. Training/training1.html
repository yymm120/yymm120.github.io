                                                                                                                                                                      Broken Access Control Access control enforce policy such that users cannot act outside of their intended permissions. Failures typically lead to unauthorized information disclosure, modification, or destruction of all data or perfoming a business fucntion outside the user's limits. 访问控制强制执行策略，以使用户不能在其预期权限之外进行操作。故障通常会导致未经授权的信息泄露、修改或破坏所有数据或执行超出用户限制的业务功能。

                                                                                                                                                                      paragraph 3: injection

                                                                                                                                                                      Some of the more common injections are SQL, NoSQL, OS command, Objet Relational Mapping(ORM), LDAP, and Expression Language(EL) or Object Graph Navigation Library (OGNL) injection. The concept is identical among all interpreters. Source code review is the best method of detection if applications are vulnerable to injections. Automated testing of all parameters, headers, URL, cookies, JSON, SOAP, and XML data inputs is strongly encouraged. Organizations can include static (SAST), dynimica(DAST), and interactive(IAST) application security testing tools into the CI/CD pipeline to identify introduced injection flaws before production deployment. 一些更常见的注入是 SQL、NoSQL、OS 命令、对象关系映射 (ORM)、LDAP 和表达式语言 (EL) 或对象图导航库 (OGNL) 注入。这个概念在所有解释器中都是相同的。如果应用程序容易受到注入攻击，源代码审查是最好的检测方法。强烈建议对所有参数、标头、URL、cookie、JSON、SOAP 和 XML 数据输入进行自动化测试。组织可以在 CI/CD 管道中包含静态 (SAST)、动态 (DAST) 和交互式 (IAST) 应用程序安全测试工具，以在生产部署之前识别引入的注入缺陷。

                                                                                                                                                                      paragraph 4: Insecure Design

                                                                                                                                                                      Insecure design is broad category representing different weaknesses, expressed as "missing or inffective control design". Insecure design is not the source for other Top 10 risk categories. There is a difference between insecure design ad insecure implementation. We differentiate between design flows and implementation defects for a reason, they have different root causes and remediation. A secure design can still have implementation defects leading to vulnerabilities that may be exploited . An insecure design cannot be fixed by a perfect implementation as by definition, needed security controls were never created to defend against specific attacks. One of the factors that contribute to insecure design is the lack of business risk profiling inherent in the software or system being developed, and thus the failure to determine what level of security design is required.

                                                                                                                                                                      不安全设计是代表不同弱点的广泛类别，表示为“缺失或感染控制设计”。不安全的设计不是其他十大风险类别的来源。不安全的设计和不安全的实施是有区别的。我们区分设计流程和实现缺陷是有原因的，它们有不同的根本原因和补救措施。一个安全的设计仍然可能存在导致可能被利用的漏洞的实现缺陷。不安全的设计无法通过完美的实现来解决，因为根据定义，从未创建过所需的安全控制来防御特定攻击。导致不安全设计的因素之一是缺乏正在开发的软件或系统中固有的业务风险分析，因此无法确定需要什么级别的安全设计。

                                                                                                                                                                      信息泄露

                                                                                                                                                                      信息泄漏在错误消息中披露信息是经常做的，因为它可以帮助普通用户修复错误，并帮助开发人员修复各种问题。可能泄露信息的错误消息示例有ODBC错误消息、身份验证错误消息等。

                                                                                                                                                                      错误处理不当不当的错误处理可以提供用于攻击系统的信息。例如，错误消息如下:“运行带有X模块的V4.1”，可以提供应用/平台识别信息。"在文件this is_a_weak_app.asp的第123行遇到的错误"可以提供应用程序实现的详细信息。“为用户id xxxxx提供的密码无效”可以透露准确的用户id。

                                                                                                                                                                      防止资讯外泄提前确定:什么是合理的，可以公开的。什么是不能透露的。从攻击者的角度思考。对所有敏感数据错误使用标准错误响应。例如“访问数据库时发生错误”。

                                                                                                                                                                      会话劫持是指更改会话数据来模拟另一个用户的会话数据，接管该用户的会话。

                                                                                                                                                                      Session劫持可以通过以下几种方式实现:

                                                                                                                                                                      1. 攻击者拦截客户端和服务器之间的通信，并窃取一个有效的会话ID。
                                                                                                                                                                      2. 攻击者试图从存储在用户计算机中的cookie中窃取会话信息。
                                                                                                                                                                      3. 猜一个可预测的会话ID

                                                                                                                                                                      防止会话ID劫持为了防止或减少Session劫持:

                                                                                                                                                                      1. 使用安全的随机会话id，如JSESSIONID。
                                                                                                                                                                      2. 将会话ID绑定到另一条标识信息，例如用户的IP地址。
                                                                                                                                                                      3. 在登出时和一段设置的不活动时间后过期会话。

                                                                                                                                                                      Establish Security Requirements

                                                                                                                                                                      During the requirements phase, you should:

                                                                                                                                                                      Assign a security advisor, either internal to your organization, if available, or from a third-party organization. The security advisor needs to be a software security expert and will be the one validating security activities and deliverables. 建立安全要求

                                                                                                                                                                      在需求阶段，您应该：

                                                                                                                                                                      分配一名安全顾问，可以是您组织内部的（如果有），也可以是来自第三方组织的。安全顾问需要是一名软件安全专家，并且是验证安全活动和可交付成果的人。 Assign security leads to the different teams involved in the process, such as the development and testing teams. The responsibility of these security leads will be to drive security efforts within their team. 将安全线索分配给流程中涉及的不同团队，例如开发和测试团队。这些安全主管的职责是推动团队内部的安全工作。 Establish a security bug bar specifying clear-cut security shipping requirements such as ALL vulnerabilities allowing an attacker to access stored or transmitted credit card numbers MUST be remediated before release. Gather security requirements applicable to your software project such as any security standards or regulations it might be subject to including PCI DSS, HIPAA, SOX, GLBA, and Basel II. Identify, acquire, and configure any software security tools needed later in the project, such as configuring your bug tracking system to track security vulnerabilities. 建立一个安全漏洞栏，指定明确的安全运输要求，例如允许攻击者访问存储或传输的信用卡号的所有漏洞，必须在发布前修复。 收集适用于您的软件项目的安全要求，例如它可能需要遵守的任何安全标准或法规，包括 PCI DSS、HIPAA、SOX、GLBA 和 Basel II。 识别、获取和配置项目后期需要的任何软件安全工具，例如配置错误跟踪系统以跟踪安全漏洞。

                                                                                                                                                                      Because it is usually not possible to address all threats at once, it is important for your organization to be able to prioritize them efficiently:

                                                                                                                                                                      Blocker - Blocks development and/or testing work, production could not run. Critical - Crashes, loss of data, severe memory leak. Major - Major loss of function. Minor - Minor loss of function, or other problem where easy workaround is present. Trivial - Cosmetic problem like misspelled words or misaligned text 由于通常不可能一次解决所有威胁，因此您的组织必须能够有效地确定它们的优先级：

                                                                                                                                                                      Blocker - 阻止开发和或测试工作，生产无法运行。 严重 - 崩溃、数据丢失、严重的内存泄漏。 严重 - 功能严重丧失。 轻微 - 轻微的功能损失，或存在简单解决方法的其他问题。 琐碎 - 诸如拼写错误的单词或未对齐的文本之类的化妆品问题 Mitigating Threats In Your Software’S Design

                                                                                                                                                                      Once application threats have been identified, categorized, and prioritized, they should be addressed by the design of your software. Threats can be mitigated by integrating appropriate countermeasures in the design. 减轻软件设计中的威胁

                                                                                                                                                                      一旦识别、分类和确定应用程序威胁的优先级，就应该通过软件设计来解决它们。通过在设计中集成适当的对策可以减轻威胁。

                                                                                                                                                                      Develop Secure Code

                                                                                                                                                                      Developers should be aware of the security peculiarities of each technology they are using to avoid introducing vulnerabilities. Often, specific technologies involve specific secure coding best practices. 开发安全代码

                                                                                                                                                                      开发人员应了解他们使用的每种技术的安全特性，以避免引入漏洞。通常，特定技术涉及特定的安全编码最佳实践。

                                                                                                                                                                      Some best practices include:

                                                                                                                                                                      Perform input and data validation Do not use insecure APIs Fail securely Protect sensitive data Manage accounts securely Implement proper authorization Follow secure auditing and logging procedures Perform manual and automated security code reviews to catch security bugs before software is sent out for testing 一些最佳实践包括：

                                                                                                                                                                      执行输入和数据验证 不要使用不安全的 API 安全失败 保护敏感数据 安全地管理帐户 实施适当的授权 遵循安全的审计和日志记录程序 在发送软件进行测试之前，执行手动和自动安全代码审查以发现安全漏洞
