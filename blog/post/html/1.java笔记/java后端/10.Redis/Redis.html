                                                                                                                                                                      NoSQL概述

                                                                                                                                                                      1.NoSLQ是什么

                                                                                                                                                                      SQL (Structured Query Language) 结构化查询语言， 泛指关系型数据库，主要代表：MySQL，Oracle，SQL Server。

                                                                                                                                                                      NoSQL（Not Only SQL），即“不仅仅是SQL”， 是一项全新的数据库理念，泛指非关系型数据库，主要代表：Redis，MongoDB。

                                                                                                                                                                      2.NoSLQ的特点

                                                                                                                                                                      • 高性能

                                                                                                                                                                        NoSQL数据库都具有非常高的读写性能，尤其在大数据量下，同样表现优秀。这得益于它的无关系性，数据库的结构简单、在内存中存取数据。

                                                                                                                                                                      • 高可用

                                                                                                                                                                        可以减少因日常维护操作（计划）和突发的系统崩溃（非计划）所导致的停机时间，以提高系统和应用的可用性。

                                                                                                                                                                      • 易扩展

                                                                                                                                                                        NoSQL数据库种类繁多，但是一个共同的特点都是去掉关系数据库的关系型特性。数据之间无关系，这样就非常容易扩展。也无形之间，在架构的层面上带来了可扩展的能力。

                                                                                                                                                                      • 数据模型简单

                                                                                                                                                                        NoSQL无需实现为要存储的数据建立字段，随时可以存储自定义的数据格式，比如可以直接存储一个简单的字符串。

                                                                                                                                                                      NoSQL并不是用来代替SQL的，目前许多项目都会选用 SQL + NoSQL 的组合方案。 关系型数据库适合存储结构化数据，如用户的帐号、地址。 非关系型数据库适合存储非结构化数据，如文章、评论。

                                                                                                                                                                      Redis的持久化

                                                                                                                                                                      Redis的高性能是由于其将所有数据都存储在了内存中，为了使Redis在重启之后仍能保证数据不丢失，需要将数据从内存中同步到硬盘(文件)中，这一过程就是持久化。 Redis支持两种方式的持久化，一种是RDB方式，一种是AOF方式。可以单独使用其中一种或将二者结合使用。

                                                                                                                                                                      1.RDB持久化

                                                                                                                                                                      默认开启，该方式将内存中数据以快照的方式写入到二进制文件中，默认的文件名为dump.rdb，可以配置时间间隔。

                                                                                                                                                                      在 redis.conf 中默认有如下配置：

                                                                                                                                                                      save 3600 1
save 300 100
save 60 10000

                                                                                                                                                                      该配置即为RDB方式数据持久化时机：

                                                                                                                                                                      关键字 时间(秒) key修改数量 解释
                                                                                                                                                                      save 3600 1 每3600秒(1小时)至少有1个key发生变化，则dump内存快照
                                                                                                                                                                      save 300 100 每300秒(5分钟)至少有100个key发生变化，则dump内存快照
                                                                                                                                                                      save 60 10000 每60秒(1分钟)至少有10000个key发生变化，则dump内存快照

                                                                                                                                                                      2.AOF持久化

                                                                                                                                                                      默认关闭，要使用时需要配置，该方式会将每一个收到的写命令都通过write函数追加到文件中，默认的文件名是appendonly.aof。

                                                                                                                                                                      开启 AOF ：在 redis.conf 文件中，将 appendonly no 改为 appendonly yes 后，重启Redis即可。

                                                                                                                                                                      在 redis.conf 中默认有如下配置：

                                                                                                                                                                      # appendfsync always
appendfsync everysec
# appendfsync no

                                                                                                                                                                      该配置即为AOF方式数据持久化时机：

                                                                                                                                                                      关键字 持久化时机 解释
                                                                                                                                                                      appendfsync always 每执行一次更新命令，持久化一次
                                                                                                                                                                      appendfsync everysec 每秒钟持久化一次
                                                                                                                                                                      appendfsync no 不持久化

                                                                                                                                                                      3.对比

                                                                                                                                                                      3.1.RDB优缺点

                                                                                                                                                                      • 优点 RDB 是一个非常紧凑的文件，它保存了 Redis 在某个时间点上的数据集，这种文件非常适合用于进行备份。 RDB 在恢复大数据集时的速度比 AOF 的恢复速度要快，因为其文件要比AOF的小。 RDB的性能要比AOF更好。
                                                                                                                                                                      • 缺点 RDB的持久化不够及时(一定时间间隔)，可能会存在数据丢失。 RDB持久化时如果文件过大可能会造成服务器的阻塞，停止客户端请求。

                                                                                                                                                                      3.2.AOF优缺点

                                                                                                                                                                      • 优点 AOF数据持久化更加耐久，可以每秒 或 每次操作保存一次。 AOF 文件有序地保存了对数据库执行的所有写入操作， 这些写入操作以 Redis 协议的格式保存， 因此 AOF 文件的内容非常容易被人读懂， 对文件进行分析也很轻松。

                                                                                                                                                                        AOF是增量操作

                                                                                                                                                                      • 缺点 对于相同的数据集来说，AOF 文件的体积通常要大于 RDB 文件的体积。 根据所使用的 appendfsync 策略，AOF 的速度可能会慢于 RDB。

                                                                                                                                                                      3.3.选择

                                                                                                                                                                      如果可以承受数分钟以内的数据丢失，选择RDB持久化。 如果对数据的完整性要求比较高, 选择AOF。

                                                                                                                                                                      Jedis连接池

                                                                                                                                                                      jedis连接资源的创建与销毁是非常消耗程序性能的，所以jedis提供了池化技术 JedisPool 。 JedisPool 在创建时会初始化一些连接资源存储到连接池中，使用jedis连接资源时不需要创建，而是从连接池中获取一个连接进行redis的操作，使用完毕后，不需要销毁该jedis连接资源，而是将该资源归还给连接池，供其他请求使用。

                                                                                                                                                                      使用方式

                                                                                                                                                                      @Test
public void test6() {
    //创建连接池的配置对象，进行一些属性的配置
    JedisPoolConfig jedisPoolConfig = new JedisPoolConfig();
    //最大连接数，默认8
    jedisPoolConfig.setMaxTotal(20);
    //最大空闲数，默认8
    jedisPoolConfig.setMaxIdle(12);
    //最大等待时间（毫秒），默认-1，表示永不超时
    jedisPoolConfig.setMaxWaitMillis(3000);

    //创建jedis连接池
    JedisPool jedisPool = new JedisPool(jedisPoolConfig, host, port, 3000, password);
    //从连接池中获取连接
    Jedis jedis = jedisPool.getResource();

    //调用jedis的方法操作redis服务器
    jedis.set("username", "王五");
    String username = jedis.get("username");
    System.out.println("获取到的数据：" + username);
    
    //归还连接到连接池中
    jedis.close();
}

                                                                                                                                                                      @Configuration
public class RedisConfig {

 /**
  * 连接池配置信息
  */
 @Bean
 public JedisPoolConfig jedisPoolConfig() {
     JedisPoolConfig jedisPoolConfig = new JedisPoolConfig();
     //最大连接数，默认8
     jedisPoolConfig.setMaxTotal(20);
     //最大空闲数，默认8
     jedisPoolConfig.setMaxIdle(12);
     //最大等待时间（毫秒），默认-1，表示永不超时
     jedisPoolConfig.setMaxWaitMillis(3000);

     return jedisPoolConfig;
 }

 /**
  * 连接工厂
  */
 @Bean
 public RedisConnectionFactory redisConnectionFactory(JedisPoolConfig jedisPoolConfig) {
     RedisStandaloneConfiguration redisStandaloneConfiguration = new RedisStandaloneConfiguration();
     //设置redis服务器的host或者ip地址
     redisStandaloneConfiguration.setHostName("192.168.1.4");
     //设置redis的服务的端口号
     redisStandaloneConfiguration.setPort(6379);
     //设置密码
     redisStandaloneConfiguration.setPassword(RedisPassword.of("123"));

     //Jedis客户端配置
     JedisClientConfiguration jedisClientConfiguration = JedisClientConfiguration
             .builder() //获取默认的连接池构造器
             .usePooling() //使用连接池
             .poolConfig(jedisPoolConfig) //加载连接池配置
             .build(); //构建客户端配置

     //jedis连接工厂
     return new JedisConnectionFactory(redisStandaloneConfiguration, jedisClientConfiguration);
 }

 /**
  * 使用Jackson序列化对象
  */
 @Bean
 public RedisSerializer<Object> redisSerializer() {
     //使用Jackson2JsonRedisSerializer来序列化和反序列化redis的value值
     Jackson2JsonRedisSerializer<Object> serializer = new Jackson2JsonRedisSerializer<>(Object.class);
     ObjectMapper mapper = new ObjectMapper();
     mapper.setVisibility(PropertyAccessor.ALL, JsonAutoDetect.Visibility.ANY);
     mapper.enableDefaultTyping(ObjectMapper.DefaultTyping.NON_FINAL);
     serializer.setObjectMapper(mapper);
     return serializer;
 }

 /**
  * RedisTemplate
  * Spring封装的Redis客户端
  */
 @Bean
 public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory factory, RedisSerializer<Object> redisSerializer) {
     RedisTemplate<String, Object> template = new RedisTemplate<>();
     //设置连接工厂
     template.setConnectionFactory(factory);

     //字符串方式序列化KEY
     StringRedisSerializer stringRedisSerializer = new StringRedisSerializer();
     template.setKeySerializer(stringRedisSerializer);
     template.setHashKeySerializer(stringRedisSerializer);

     //JSON方式序列化VALUE
     template.setValueSerializer(redisSerializer);
     template.setHashValueSerializer(redisSerializer);

     template.afterPropertiesSet();
     return template;
 }
}

                                                                                                                                                                      案例：使用Redis缓存数据

                                                                                                                                                                      1.缓存角色列表数据

                                                                                                                                                                      @Service
public class RoleServiceImpl implements RoleService {

    @Resource
    private RoleMapper roleMapper;

    @Resource
    private RedisTemplate<String, Object> redisTemplate;

    @Override
    public List<Role> getRoleList() {
        List<Role> roleList;
        if (redisTemplate.hasKey("roleList")) {
            //Redis中有，则取Redis中的缓存
            roleList = (List<Role>) redisTemplate.opsForValue().get("roleList");
        } else {
            //Redis中没有，则查询数据库后，存储到Redis中
            roleList = roleMapper.getRoleList();
            redisTemplate.opsForValue().set("roleList", roleList, Duration.ofHours(12));
        }

        return roleList;
    }
}

                                                                                                                                                                      2.缓存用户列表

                                                                                                                                                                      @Service
public class UserServiceImpl implements UserService {
    @Resource
    private UserMapper userMapper;

    @Resource
    private RedisTemplate<String, Object> redisTemplate;
    
    @Override
    public List<User> getUserList(String userName, Integer userRole, Integer start, Integer pageSize){
        List<User> userList;

        //使用查询条件组成key
        String key = "User:getUserList:" + userName + ":" + userRole + ":" + start + ":" + pageSize;
        if (redisTemplate.hasKey(key)) {
            //Redis中有，则取Redis中的缓存
            userList = (List<User>) redisTemplate.opsForValue().get(key);
        } else {
            //Redis中没有，则查询数据库后，存储到Redis中
            userList = userMapper.getUserList(userName, userRole, start, pageSize);
            redisTemplate.opsForValue().set(key, userList, Duration.ofHours(6));
        }

        return userList;
    }
}
