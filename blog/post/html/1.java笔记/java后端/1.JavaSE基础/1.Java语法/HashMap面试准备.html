                                                                                                                                                                      简述hashMap的put()方法？

                                                                                                                                                                      HashMap什么时候扩容？

                                                                                                                                                                      答： hashMap在以下两种情况下会进行扩容

                                                                                                                                                                      • 一、hashMap中桶的数量超过了阈值。
                                                                                                                                                                      • 二、单链表转为红黑树时，若数组容量小于64，会进行扩容。

                                                                                                                                                                      红黑树需要左旋、右旋和变色来保持自身的平衡，在数组比较小的时候，选择扩容比选择将链表转为红黑树更划算。

                                                                                                                                                                      HashMap如何扩容？

                                                                                                                                                                      答： hashMap扩容使用resize方法，这个方法既用来初始化，也被用来扩容。

                                                                                                                                                                      • 一、当进入resize方法后，会判断应该执行初始化还是应该执行扩容。
                                                                                                                                                                      • 如果执行初始化，则执行设置容量、设置阈值、创建数组这三个操作。容量和阈值默认值是16和0.75。
                                                                                                                                                                      • 如果是扩容，在判断容量不超过最大值后，执行左移容量、左移阈值、复制数组这三个操作。左移时，会将容量和阈值均左移一位，也就是容量翻倍。而复制数组时，jdk7会遍历所有元素的同时重新计算hash，这个过程非常耗时，jdk8不需要重新计算hash，这得益于jdk8计算hash采用的二次扰动公式。也就是index(hash)等于$(length-1) & hashCode$ 。

                                                                                                                                                                      如果是JDK8，遍历元素不会重新计算index，新的位置要么就是原来的位置，要么index=原来的位置+旧容量。（index=(length-1)&hashCode，length是2次幂，length-1始终全为1，当向左移一位，与原来相比不过多了一个1。hashCode若高位为0，与运算后，index位置不变。hashCode若高位为1，与运算后，index位置的最高位写1。）

                                                                                                                                                                      所以，JDK8的hashMap扩容之后，存储位置要么是原位置、要么是原位置+旧容量这个位置。

                                                                                                                                                                      当然，前提是length必须是2的幂次。

                                                                                                                                                                      简述hashMap的hash()方法？

                                                                                                                                                                      ​ hashMap中的hash增加了扰动函数，降低了hash冲突的几率。jdk8进行了两次扰动运算：

                                                                                                                                                                      1. hashCode ^ (hashCode >>>16)
                                                                                                                                                                      2. hash&(length-1)
                                                                                                                                                                      static final int hash(Object key) {
    int h;
    // 与自己右移16位进行异或运算（高低位异或）
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}

                                                                                                                                                                      为什么将hashcode右移16位？

                                                                                                                                                                      ​ hash = hashCode ^ (hashCode >>> 16)

                                                                                                                                                                      原因：

                                                                                                                                                                      		1. 为什么要右移？计算出的hashCode是int型32位。可能出现高位不同，但低位相同的hashCode，比如低位全是0的hashCode。在进行`hash&(length-1)`运算的时候，只要length-1的长度不超过低位部分相同的长度，那结果与运算的完全相等。
		2. 为什么要右移16位？因为hashCode是32位的，如果低位相同，但高位不同。那只需要右移16位，然后再亦或，就可以让高位也加入运算中。（亦或：相同为0不同为1）

                                                                                                                                                                      为什么length为2的次幂？

                                                                                                                                                                      答：1. hashMap需要较大的数组容量保证hash的离散性。 2.二进制运算效率比取模运算效率高。 3.length为2的次幂，扩容后length左移一位，元素在新数组中的位置可直接确定，不需要重新计算。

                                                                                                                                                                      length为2的次幂，length-1始终全为1，length扩容左移一位，length最高位为1，length与上hash，最高位可能为0可能为1。取决于hash的对应位上的值是0还是1。扩容之后的位置要么还在原位、要么原位+原来的length。

                                                                                                                                                                      单向链表

                                                                                                                                                                      线性结构，查找、索引、增、删都是O(n)，头节点的操作为O(1)。

                                                                                                                                                                      红黑树

                                                                                                                                                                      红黑树是二叉查找树，是自平衡的二叉查找树。但不是完美二叉查找树。树的高度是有可能很大的。

                                                                                                                                                                      定义和性质

                                                                                                                                                                      • 性质1：每个节点要么黑色、要么红色。
                                                                                                                                                                      • 性质2：根节点是黑色。
                                                                                                                                                                      • 性质3：叶子节点是黑色。
                                                                                                                                                                      • 性质4：红节点的子节点全是黑色。黑节点则不一定。
                                                                                                                                                                      • 性质5：任意节点到任意叶子节点，都包含数量相同的黑节点。

                                                                                                                                                                      红黑树每个节点左边一定比它小，右边一定比它大。

                                                                                                                                                                      红黑树的自平衡

                                                                                                                                                                      红黑树的自平衡依靠三种操作：左旋、右旋、变色。

                                                                                                                                                                      左旋：把某节点的（右）子节点变为根节点。

                                                                                                                                                                      右旋：把某节点的（左）子节点变为根节点。

                                                                                                                                                                      变色：红节点的子节点一定是黑色。

                                                                                                                                                                      常见hashmap面试问题

                                                                                                                                                                      HashMap与HashTable的区别？

                                                                                                                                                                      答：

                                                                                                                                                                      1. HashMap是线程不安全的，HashTable是线程安全的。
                                                                                                                                                                      2. HashMap允许Key和Value都为null，HashTable都不允许为null。
                                                                                                                                                                      3. HashMap继承自AbstractMap类，而HashTable继承自Dictionary。

                                                                                                                                                                      为什么链表个数超过8才转为红黑树？

                                                                                                                                                                      答：红黑树的节点在内存中比链表节点大两倍，所以优先使用链表。至于为什么要超过８才转红黑树，这是因为数据均匀的分布在数组上，链表长度到达8的概率为0.00000006，几乎不会有链表转为红黑树。之所以选择8是因为链表长度超过8的概率已经很小了。

                                                                                                                                                                      为什么红黑树的个数是小于6时转为链表？

                                                                                                                                                                      答：避免频繁的红黑树与链表的转换

                                                                                                                                                                      String中的hashCode

                                                                                                                                                                      ​ hashCode与hashMap中的hash不同，hashMap中的key需要通过hashCode来判断不同，需要通过eqals判断相同。String中重写了hashCode和equal方法。

                                                                                                                                                                      public int hashCode() {
    int h = hash;
    if (h == 0 && value.length > 0) {
        char val[] = value;

        for (int i = 0; i < value.length; i++) {
            h = 31 * h + val[i];
        }
        hash = h;
    }
    return h;
}

                                                                                                                                                                      String类中的hashCode计算方法还是比较简单的，就是以31为权，每一位为字符的ASCII值进行运算，用自然溢出来等效取模。

                                                                                                                                                                      哈希计算公式可以计为s[0]*31^(n-1) + s[1]*31^(n-2) + ... + s[n-1]

                                                                                                                                                                      那为什么以31为质数呢?

                                                                                                                                                                      主要是因为31是一个奇质数，所以31*i=32*i-i=(i<<5)-i，这种位移与减法结合的计算相比一般的运算快很多。

                                                                                                                                                                      HashMap 7到8有哪些改动

                                                                                                                                                                      ​

                                                                                                                                                                      ​ JDK7到JDK8的改动中，有几个重大的改动：

                                                                                                                                                                      1. JDK7是数组加单向链表，JDK8是数组加单向链表加红黑树，主要是为了避免极端情况下，单链表的查找速度太慢，不稳定；
                                                                                                                                                                      2. JDK7中的单链表采用的头插法，JDK8是尾插法，在扩容的时候，头插法可能出现链表反转，在多线程的情况下，get取值可能导致死循环，。
                                                                                                                                                                      3. JDK7的计算hash时和JDK8不同，JDK7进行了9次扰动，但是JDK8计算hash进行了两次扰动，扰动其实就是减少hash冲突的一种计算，jdk8的两次扰动，一次是通过向右移位再亦或，让高位部分也加入计算。另一次是与运算，也就是计算数组index的计算。好处是JDK8这样的两次扰动，可以直接判断出扩容后元素的index，要么在原来的位置，要么在index等于原来的位置+原来的容量。完全不需要重算index。
                                                                                                                                                                      4. 第四个改动是，JDK7是先扩容再插入，JDK8是先插入再扩容。 这个改动主要是因为扩容的开销太大了。如果是JDK7，判断一下负载是否到75%，就可以扩容。但是JDK8不想频繁扩容，所以它先插进去，看看是插在数组上还是插在链表和红黑树上，如果是没有插在数组上，那就不需要扩容。

                                                                                                                                                                      JDK1.7中HashMap出现死循环的原因

                                                                                                                                                                      put可能导致扩容操作，且会重新计算hash。多线程下，如果其他线程put时，hash值与之前重新计算的hash值相同，就会分配到同一个链表下，且它们的下一个节点都可能指向对方。可能做成闭环，导致在get时，出现死循环。

                                                                                                                                                                      其他的Map的使用场景

                                                                                                                                                                      类 介绍 使用场景
                                                                                                                                                                      HashTable 早期的线程安全Map，直接在方法中加synchronized。 理论上不会使用
                                                                                                                                                                      ConcurrentHashMap 线程安全的Map，通过synchronized + CAS实现 需要保证线程安全
                                                                                                                                                                      LinkedHashMap 能记录访问顺序或插入顺序的Map，通过head、tail属性维护有序双向链表，通过Entry的after、before属性维护节点的顺序 需要记录访问顺序或插入顺序

                                                                                                                                                                      通过volatile、final、CAS等lock-free技术减少锁竞争对于性能的影响。

                                                                                                                                                                      CAS是什么？

                                                                                                                                                                      CAS是英文单词CompareAndSwap的缩写，中文意思是：比较并替换。CAS需要有3个操作数：

                                                                                                                                                                      ConcurentHashMap1.7与1.8的分别的实现原理

                                                                                                                                                                      • JDK1.7的ConcurrentHashMap采用分段锁Segment

                                                                                                                                                                      Concurrent中的分段所称为segment，将数据分成一段一段的存储，然后给每一段的数据配一把锁，当一个线程占用其中一段数据的时候，其他段的数据可以被其他线程访问。

                                                                                                                                                                      ｓｅｇｍｅｎｔ继承自ReenTrantLock类

                                                                                                                                                                      缺点：在hash的时候，要先定位到segment、再定位该段中的元素。

                                                                                                                                                                      • JDK1.8的ConcurrentHashMap采用CAS结合synchronized。

                                                                                                                                                                      　　CAS（Compare-and-Swap），即比较并替换，是一种实现并发算法时常用到的技术。CAS有三个操作数，内存地址V，旧的预期值A，新的预期值B。当CAS指令执行时，当且仅当内存地址V与旧预期值A相等，将内存地址V的值修改为B，否则旧什么都不做。（整个比较并替换的操作就是一个原子操作）

                                                                                                                                                                      1.7put写操作：先确定segment、再通过segment的put方法确定要写入的位置。

                                                                                                                                                                      1.7get读操作：获取指定key对应的value，不需要获取锁。

                                                                                                                                                                      1.7size容量：计算各个segment的table数组内所有链表的节点个数。首先不加锁的计算两次，如果结果相同则直接返回。如果结果不同，则再加锁计算一次。

                                                                                                                                                                      1.8put写操作：实现逻辑与hashMap的putVal一致。put时，链表头节点的操作用硬件级别的CAS操作。不是头节点，则使用synchronized来同步多个线程对该链表的访问。

                                                                                                                                                                      为什么不是头节点，需要用synchronized来同步？

                                                                                                                                                                      答：如果不是头节点，则需要通过遍历才能确定节点的位置，遍历的过程中，其他线程有可能会添加这个节点。故而通过synchronized锁住链表头节点保证任何时候只有一个线程对链表进行更新

                                                                                                                                                                      1.8的get读操作：整个过程不需要加锁。

                                                                                                                                                                      1.8size容量：通过baseCount和CounterCell累加实现，当线程每次增删后，线程尝试向baseCount更新，如果失败，则将数字累加到countercell中，当调用size方法时，会累加counterCells数组和baseCount数组。在整个size()方法中不需要加锁。

                                                                                                                                                                      ConcurrentHashMap和Hashtable的区别？

                                                                                                                                                                      答：ConcurrentHashMap 结合了 HashMap 和 HashTable 二者的优势。HashMap 没有考虑同步，HashTable 考虑了同步的问题。但是 HashTable 在每次同步执行时都要锁住整个结构。 ConcurrentHashMap 锁的方式是稍微细粒度的。

                                                                                                                                                                      Java集合的快速失败机制 “fail-fast”？

                                                                                                                                                                      答：

                                                                                                                                                                      是java集合的一种错误检测机制，当多个线程对集合进行结构上的改变的操作时，有可能会产生 fail-fast 机制。

                                                                                                                                                                      例如：假设存在两个线程（线程1、线程2），线程1通过Iterator在遍历集合A中的元素，在某个时候线程2修改了集合A的结构（是结构上面的修改，而不是简单的修改集合元素的内容），那么这个时候程序就会抛出 ConcurrentModificationException 异常，从而产生fail-fast机制。

                                                                                                                                                                      原因：迭代器在遍历时直接访问集合中的内容，并且在遍历过程中使用一个 modCount 变量。集合在被遍历期间如果内容发生变化，就会改变modCount的值。每当迭代器使用hashNext()/next()遍历下一个元素之前，都会检测modCount变量是否为expectedmodCount值，是的话就返回遍历；否则抛出异常，终止遍历。

                                                                                                                                                                      解决办法：

                                                                                                                                                                      1. 在遍历过程中，所有涉及到改变modCount值得地方全部加上synchronized。

                                                                                                                                                                      2. 使用CopyOnWriteArrayList来替换ArrayList

                                                                                                                                                                      HashSet是如何保证数据不可重复的？

                                                                                                                                                                      答：HashSet的底层其实就是HashMap，只不过我们HashSet是实现了Set接口并且把数据作为K值，而V值一直使用一个相同的虚值来保存，我们可以看到源码：

                                                                                                                                                                      public boolean add(E e) {
    return map.put(e, PRESENT)==null;// 调用HashMap的put方法,PRESENT是一个至始至终都相同的虚值
}

                                                                                                                                                                      由于HashMap的K值本身就不允许重复，并且在HashMap中如果K/V相同时，会用新的V覆盖掉旧的V，然后返回旧的V，那么在HashSet中执行这一句话始终会返回一个false，导致插入失败，这样就保证了数据的不可重复性；

                                                                                                                                                                      ConcurrentHashMap的具体实现？

                                                                                                                                                                      答：在JDK1.7中，ConcurrentHashMap采用Segment + HashEntry的方式进行实现，结构如下：

                                                                                                                                                                      1. 该类包含两个静态内部类 HashEntry 和 Segment ；前者用来封装映射表的键值对，后者用来充当锁的角色；
                                                                                                                                                                      2. Segment 是一种可重入的锁 ReentrantLock，每个 Segment 守护一个HashEntry 数组里得元素，当对 HashEntry 数组的数据进行修改时，必须首先获得对应的 Segment 锁。

                                                                                                                                                                      在JDK1.8中，放弃了Segment臃肿的设计，取而代之的是采用Node + CAS + Synchronized来保证并发安全进行实现，结构如下：

                                                                                                                                                                      插入元素过程（建议去看看源码）：

                                                                                                                                                                      1. 如果相应位置的Node还没有初始化，则调用CAS插入相应的数据；

    if (casTabAt(tab, i, null, new Node<K,V>(hash, key, value, null)))
        break;                   // no lock when adding to empty bin}

                                                                                                                                                                      2. 如果插入的是一个新节点，则执行addCount()方法尝试更新元素个数baseCount；

                                                                                                                                                                      针对问题二，记住如何写一个不可变类

                                                                                                                                                                      (1)类添加final修饰符，保证类不被继承。 如果类可以被继承会破坏类的不可变性机制，只要继承类覆盖父类的方法并且继承类可以改变成员变量值，那么一旦子类以父类的形式出现时，不能保证当前类是否可变。

                                                                                                                                                                      (2)保证所有成员变量必须私有，并且加上final修饰 通过这种方式保证成员变量不可改变。但只做到这一步还不够，因为如果是对象成员变量有可能再外部改变其值。所以第4点弥补这个不足。

                                                                                                                                                                      (3)不提供改变成员变量的方法，包括setter 避免通过其他接口改变成员变量的值，破坏不可变特性。

                                                                                                                                                                      (4)通过构造器初始化所有成员，进行深拷贝(deep copy) 如果构造器传入的对象直接赋值给成员变量，还是可以通过对传入对象的修改进而导致改变内部变量的值。例如：

                                                                                                                                                                      public final class ImmutableDemo {  
    private final int[] myArray;  
    public ImmutableDemo(int[] array) {  
        this.myArray = array; // wrong  
    }  
}

                                                                                                                                                                      这种方式不能保证不可变性，myArray和array指向同一块内存地址，用户可以在ImmutableDemo之外通过修改array对象的值来改变myArray内部的值。 为了保证内部的值不被修改，可以采用深度copy来创建一个新内存保存传入的值。正确做法：

                                                                                                                                                                      public final class MyImmutableDemo {  
    private final int[] myArray;  
    public MyImmutableDemo(int[] array) {  
        this.myArray = array.clone();   
    }   
}

                                                                                                                                                                      (5)在getter方法中，不要直接返回对象本身，而是克隆对象，并返回对象的拷贝 这种做法也是防止对象外泄，防止通过getter获得内部可变成员对象后对成员变量直接操作，导致成员变量发生改变。
